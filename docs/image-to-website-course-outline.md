# 从一张图到前端网页：流程教程

## 核心观点

很多人会用 AI 去还原 GPT 生成的官网效果图，但结果通常不理想。原因是：一张图里包含了文案、布局、配图、间距、响应式、资源管理和上线验证，直接让 AI 一次性生成 HTML，很容易只得到一个“看起来像”的页面。

这节课要讲的，就是如何把一张完整官网参考图，转化为可维护、可响应式、可发布的真实前端网页。重点放在整个底层逻辑和工作流：怎么和 Codex 对话，怎么留存事实源，怎么分步实现，怎么用截图纠偏，最后怎么验证和发布。

## 1. 先定义任务和工程约束

真实对话：

> 我希望100%还原这个图，是一个产品官网，以html开发，但是，现在内容，很多且只有图，我们先讨论一下如何做？是否需要把图片分区，逐步还原，还有其中的配图部分，需要重新生成高清的。但是风格和样式要保持一致。我们来讨论一个合理的还原方案。

> 完全照这张图做一个独立官网，然后网页应该是响应式的而不是固定尺寸，这个也很重要。

方法论：

- 第一句话先定任务性质：官网还原、HTML 实现、风格一致。
- 第二句话提出风险：内容多、只有图、配图需要高清重做。
- 第三句话补工程要求：独立官网、响应式、不能固定尺寸。
- 这会让 Codex 先判断架构，再进入实现。

Codex 动作：

- 先拆页面模块。
- 再判断哪些用代码实现，哪些做成图片资产。
- 最后给出从图到网页的实现路径。

对应素材：

- 对话截图：最初提出“100% 还原 + HTML + 先讨论方案”的截图。
- 执行结果图：Codex 拆解页面结构和响应式策略的回复截图。

## 2. 固化事实源，AI 切错时让用户接管

真实对话：

> 我们需要先做规划，然后留存一些信息，要不上下文丢失或者压缩了，是不是就没有了，这个图应该改作为唯一事实源留存。

> 还有一个问题，你觉得需要分区块，去截图分别留存吗？一整个网页，你的读取信息量能不能承受住？

> 不行，你切的完全不对，这样吧，我来切。然后还是按照现在的命名来。

> 序号是按照从上到下的区块排列的，你来修改名称。然后分区可能和你规划的分区不一样，需要你根据这些图，再重新调整md内容。

方法论：

- 先固定事实源：把原图留在项目里，避免上下文丢失。
- 再补局部证据：用分区切片解决细节读取问题。
- 发现风险就接管：AI 切错时，用户直接提供正确切片。
- 最后同步文档：让 Codex 更新“切片目录表”和还原规划。

Codex 执行结果：

- 保存完整原图，作为后续还原的最高参考。
- 建立一份还原规划，记录页面分区、资产需求和验收标准。
- 用用户手工切片替换 AI 自动切错的版本。
- 更新切片目录表 `manifest.md`：说明每张切片对应页面里的哪个模块，后续开发时按它查找参考图。

对应素材：

- 对话截图：要求“唯一事实源”和“我来切”的两段对话。
- 执行结果图：原图、7 张切片、`manifest.md` 切片目录表的文件截图。

## 3. 先搭可替换网页框架

真实对话：

> 然后是这样的，需要重新生成的配图，我会自己用gpt去做。我需要你做的，是按照这个图，把框架搭好，然后后续直接把图做替换，你能做吗。

方法论：

- 先搭网页框架：不用等所有高清配图完成。
- 先做可编辑内容：文案、按钮、卡片、导航、Footer 都用 HTML/CSS。
- 先留稳定资产位：复杂图片用占位图撑住结构。
- 后续替换图片：覆盖同名文件，页面就能更新。

Codex 执行结果：

- 创建网页的三个基础文件：结构、样式和交互。
- 预留一组稳定图片文件名，方便后续直接替换。
- 生成同名占位图，避免页面出现破图和 404。
- 启动本地服务，验证桌面、移动端、控制台错误和横向溢出。

对应素材：

- 对话截图：用户明确“配图我自己做，你先搭框架”的截图。
- 执行结果图：项目文件结构 + 本地浏览器首版页面。

## 4. 替换 GPT 生成的局部配图

真实对话：

> 这个图里面的4个图标，分别输出独立的高清图，背景也是纯色。将宽高比设为 1:1

> 我把指标区域的四个图标放到 imagesw 文件了，目前是数字序号，你来替换一下。

方法论：

- 先让网页框架预留稳定资产名：页面结构先稳定，图片后替换。
- 再用 GPT 生成局部高清图：不要让 AI 一次重做整页，只处理复杂的视觉资产。
- 文件命名可以先简单：`01`、`02`、`03`、`04` 这种编号，也能降低沟通成本。
- Codex 负责做资产映射和页面验证：确认每张图对应哪个模块，替换后检查桌面端和移动端有没有撑乱布局。

Codex 执行结果：

- 读取用户放进图片目录的 4 张编号图。
- 按页面顺序映射为采购响应、成本降低、核算周期、BIM 预测四个指标图标。
- 覆盖到网页里原本预留的稳定图片名，HTML 路径不用反复改。
- 用浏览器验证桌面端和移动端：没有数字重叠、没有横向滚动，图片没有撑开卡片。

对应素材：

- 对话截图：GPT 生成 4 张独立图标的截图 + Codex 执行替换的截图。
- 执行结果图：指标卡区域最终页面截图，或图片目录里的 `metric-*.png` 文件。

## 5. 用品牌资产和开源图标统一视觉语言

真实对话：

> 图标部分，我觉得别自己重绘，你找一个开源的图标库，和配图差不多的。

> 蓝紫色的按钮配色用这个，网页的配色，尽量和这个统一。设计稿里的一些白色和毛玻璃效果也参考，做网站的卡片样式。字体颜色也改一下，还有现在辅助字体不要加粗。logo我直接导出了，你替换吧。

方法论：

- 先给锚点来源：包括自己的品牌资产，也包括 Codex 能检索到的开源参考。
- 自己的资产要可用：私有设计稿要补截图，logo、色值、图形最好直接导出。
- 开源参考要明确：图标库、动效库、组件风格都可以作为可落地参照。
- 最后沉淀规则：把颜色、阴影、图标风格变成全站规则。

Codex 执行结果：

- 替换 `logo.svg`。
- 用蓝紫渐变统一按钮、标题强调、图标渐变和线条。
- 把图标切换成本地 Lucide SVG sprite。
- 调整阴影为浅紫色、低透明度，贴近整体风格。

对应素材：

- 对话截图：Figma/logo/按钮色/开源图标库这几段。
- 执行结果图：logo 替换、按钮渐变、SVG sprite 文件、页面局部风格对比。

## 6. 用典型纠偏推动网页变精细

网页还原的质量，往往由一轮轮截图反馈推上去。这里选 4 个典型纠偏案例，分别对应密度、视频、移动端和动效。

### 案例 A：间距和密度

真实对话：

> 我们来调整一下，这些卡片的内边距、间距，尽量对齐、减少换行；有的间距太大了。

> 1: 1280的容器你不应该改，要求全栈一致 2: 我感觉现在整个网站的密度和字号、元素大小都不太对，对于网站和移动端都有点大了，是不是被截图影响了，你整体调整一下。

方法论：

- 先指出问题感受：别只说“不好看”，要说清楚哪里松散。
- 再点名具体参数：内边距、间距、换行、字号、元素大小。
- 发现偏航就拉回：Codex 为了修局部破坏全局规则时，要及时纠正。
- 最后给硬约束：全站保持 `1280px` 容器一致。

Codex 处理：

- 先回退破坏全站一致性的局部容器宽度。
- 再统一压缩字号、卡片内边距、图片尺寸和列表行距。
- 最后用浏览器检查容器宽度、卡片高度和横向溢出。

### 案例 B：Hero 视频融合

真实对话：

> 右侧这里是个占位图，我做了一个mp4的视频，但是是纯色背景，能做成叠加效果，或者其他方式，和背景融合吗？此外mp4会不会让页面很卡？你来替换一下试试。

> 1: 感觉叠加效果不好，还是能够看到，尤其是右上角，是不是背景的原因？

> 我觉得不行，感觉整个画面特别暗。

方法论：

- 先说视觉目标：视频要和背景融合。
- 同时提出性能担忧：MP4 不能让页面变卡。
- 先试低成本方案：叠加、混合、poster、压缩。
- 发现素材问题再升级：背景仍然露出时，改成透明 WebM。

Codex 处理：

- 先检查原 MP4 的尺寸、码率和体积。
- 再压缩视频，加入 poster，降低首屏负担。
- 第一版用混合模式融合背景；失败后改成透明 WebM。
- 透明后画面变暗，再补亮度、饱和度和轻微背光。

### 案例 C：移动端独立重排

真实对话：

> 移动端 1:气泡没有显示完整，有些丢失了 2: 五大模块，改为1行2个 3:这里指标也是一行2个 ，3:下面的时间轴，能不能改成纵向显示，在移动端 4:最底部的太长了，改为3列显示。

方法论：

- 先拆移动端问题：不要只说“适配一下”。
- 再列布局规则：气泡、模块、指标、时间轴、Footer 分别怎么排。
- 限定修改范围：只改移动端断点，避免影响桌面端。
- 最后真实验证：用 `390px` 视口检查横向溢出。

Codex 处理：

- 只改移动端断点，保留桌面布局。
- 把模块卡、指标卡、Footer 按用户要求重排。
- 时间轴先试纵向，再调整成更紧凑的移动端形式。
- 用真实移动视口检查菜单行高、横向溢出和模块显示完整度。

### 案例 D：动效怎么聊出来

动效沟通可以分成几种情况：不知道要什么、知道要什么，以及看到效果后继续纠偏。

**1. 不知道要什么：先让 AI 出方案**

真实对话：

> 我们分别讨论一下 每个区块的动画吧，比如hero区域的文字和按钮，可以怎么动

方法论：先让 AI 按区块给方案。  
不知道具体要什么时，不要只说“加点动效”，可以让 AI 分别说明 Hero、指标区、架构区、案例区、白皮书、CTA 适合怎么动。方案里最好包含触发方式、适用区块和优先级，方便后面选择和删减。

**2. 知道要什么：给参考或说动作**

真实对话：

> 参考 reactbits 或者 motion 的动画

> 某个卡片悬浮，有渐变的描边，描边最好有动效，比如环绕的效果，卡片也可以有鼠标跟随的浅色背景渐变。

> 几个气泡默认状态，也加一点上下浮动的动画吧。

方法论：知道要什么时，可以给参考，也可以直接说动作。  
动效库、网站或组件是风格锚点，可以帮助 AI 判断目标质感；具体动作要尽量说清对象、触发、变化和默认状态。比如卡片 hover 时出现渐变描边和鼠标跟随光斑，气泡默认轻微上下浮动。描述越具体，AI 越容易写成稳定的 CSS/JS。

**3. 看到效果后：继续收窄边界**

真实对话：

> 不是要做单独加载动画，是页面刷新的时候，网页的每个元素。

> 悬浮的气泡都错位了，然后我不想要加载动画了，或者我们单独去做每个模块的动画吧，现在很粗糙。

> 项目和白皮书就不要悬浮样式了。

方法论：动效做偏时，直接修正类型和范围。  
如果 AI 把“加载动画”理解成全屏 loading，就改成“页面元素进入”；如果气泡错位，就说明这个元素不适合套通用 hover；如果案例和白皮书不该动，就把动效限制在真正需要交互反馈的模块上。

Codex 处理：

- 先把 ReactBits / Motion 当成风格参照，提炼出旋转渐变描边、鼠标跟随光斑、卡片 hover、滚动进入这些可落地动作。
- 在动手前给出区块级方案：Hero 拆成文字入场、按钮 hover、主视觉呼吸、气泡浮动；后续模块按指标、架构、案例、白皮书、CTA 分别设计，不套同一种动画。
- 第一次实现包含全屏 loading 层，收到纠偏后撤掉遮罩式 loading，改成导航、Hero 文案、按钮、主视觉、卡片按层级进入。
- 发现气泡和标签原本依赖定位后，避免用 hover 动效覆盖原来的 transform，改成不破坏定位的动效叠加方式。
- 把动效范围从“全站元素都动”收敛为“适合交互的卡片动，不适合 hover 的案例区、白皮书区保持稳定”。
- 给 Hero 气泡加默认上下浮动，保留主视觉活跃度；同时用 reduced motion 和移动端策略避免动效拖慢页面。

对应素材：

- 对话截图：第一张放“ReactBits / Motion + loading 动画”的原始需求，第二张放“不是单独加载动画，是页面刷新时每个元素”的纠偏。
- 方案截图：放 Codex 对 Hero、指标区、架构区、案例区等模块的动效方案描述。
- Codex 执行结果图：放 hover 卡片、渐变描边、鼠标跟随高光、气泡上下浮动或动效代码片段。
- 讲解重点：让学员看到动效不是一句“加高级动画”就结束，而是一轮轮把感觉、动作、范围和验收标准说清楚。

## 7. 压缩资源，发布到 GitHub Pages 并做兼容回修

真实对话：

> 网站这些配图，根本不好显示，加载很慢。你看看怎么处理。

> 这两个比较大的3d图，加载很慢。

> 整理一下项目文件，吧用不到的配图都删掉，不要占用文件大小，然后在github帮我建一个公开仓库。

> 我发现在有的浏览器里，这个图标背景就会有问题，没有融合，谷歌就还行。

> css兼容性问题、不同浏览器内核版本不一致，让codex 考虑兼容性，做兼容性测试。

方法论：

- 先处理资源体积：图片不好显示、加载慢，通常不是页面结构问题，而是大图、视频、格式和 fallback 没处理好。
- 再做上线收口：删掉无用素材和过程文件，避免仓库臃肿，也避免把参考图、临时截图一起发布出去。
- 然后发布到公开链接：GitHub Pages 的价值是让页面进入真实访问环境，而不只是在本地浏览器里看起来正常。
- 发布后做兼容验收：线上、微信浏览器、Safari、Chrome 可能表现不同，真实问题要反推代码、资源格式和 fallback 策略。
- 最后回修并复测：性能、兼容性和发布不是三个独立步骤，而是上线前后一轮完整闭环。

Codex 执行结果：

- 排查图片资源体积和页面引用关系。
- 压缩图片资源，补 WebP、小尺寸图和移动端版本。
- 处理图片 fallback、视频播放按钮和移动端资源加载问题。
- 交付前先收口：删掉无用素材，避免仓库变臃肿。
- 补项目说明文档和忽略清单。
- 初始化 GitHub 仓库并发布 GitHub Pages。
- 验证线上页面、资源路径和访问状态。
- 处理 WebView 背景融合、浏览器内核差异和图标背景显示问题。

对应素材：

- 对话截图：图片加载慢、3D 图资源太大、要求清理项目、建立公开 GitHub 仓库、反馈线上兼容问题。
- 执行结果图：资源体积优化、fallback 代码、GitHub 仓库、Pages 设置、线上页面、不同浏览器验证截图。

## 8. 最终流程总结

1. 先定义任务和工程约束：说明这是官网还原、HTML 实现、响应式网页。
2. 固化事实源：保存完整原图，用分区切片和切片目录表解决细节读取问题。
3. 先搭可替换网页框架：文案、布局、按钮、卡片先落成 HTML/CSS/JS，复杂配图预留替换位。
4. 替换 GPT 生成的局部配图：用户用 GPT 生成关键高清图，Codex 负责映射、替换和页面验证。
5. 用品牌资产和开源图标统一视觉语言：用 logo、色值、图标库和视觉参考统一全站风格。
6. 用典型纠偏推动网页变精细：围绕间距密度、Hero 视频、移动端重排和动效做代表性修正。
7. 压缩资源，发布到 GitHub Pages 并做兼容回修：压缩图片和视频，补 fallback，清理项目，发布 GitHub Pages，并在真实环境里做兼容验证。

- 切片目录表：`manifest.md`，用于说明每张切片对应哪个页面模块。
- 实施规划：记录页面分区、资产需求、响应式和验收标准。
- 最终网页文件：结构、样式和交互代码。
- 图片资产：后续替换用的高清配图。
- 视频资产：Hero 区域使用的视频和兜底资源。
